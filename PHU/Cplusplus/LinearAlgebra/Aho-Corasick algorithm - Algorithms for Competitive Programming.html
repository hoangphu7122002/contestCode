    Aho-Corasick algorithm

    The Aho-Corasick algorithm allows us to quickly search for multiple patterns in a text. The set of pattern strings is also called a dictionary. We will denote the total length of its constituent strings by m<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi></math>$m$ and the size of the alphabet by k<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>k</mi></math>$k$. The algorithm constructs a finite state automaton based on a trie in O(mk)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>m</mi><mi>k</mi><mo stretchy="false">)</mo></math>$O(m k)$ time and then uses it to process the text.

    The algorithm was proposed by Alfred Aho and Margaret Corasick in 1975.

    Construction of the trie


    A trie based on words "Java", "Rad", "Rand", "Rau", "Raum" and "Rose".
    The image by nd is distributed under CC BY-SA 3.0 license.

    Formally, a trie is a rooted tree, where each edge of the tree is labeled with some letter and outgoing edges of a vertex have distinct labels.

    We will identify each vertex in the trie with the string formed by the labels on the path from the root to that vertex.

    Each vertex will also have a flag output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ which will be set if the vertex corresponds to a pattern in the dictionary.

    Accordingly, a trie for a set of strings is a trie such that each output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertex corresponds to one string from the set, and conversely, each string of the set corresponds to one output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertex.

    We now describe how to construct a trie for a given set of strings in linear time with respect to their total length.

    We introduce a structure for the vertices of the tree: 

    const int K = 26;

struct Vertex {
    int next[K];
    bool output = false;

    Vertex() {
        fill(begin(next), end(next), -1);
    }
};

vector<Vertex> trie(1);

    Here, we store the trie as an array of Vertex<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>Vertex</mtext></math>$\text{Vertex}$. Each Vertex<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>Vertex</mtext></math>$\text{Vertex}$ contains the flag output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ and the edges in the form of an array next[]<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>next</mtext><mo stretchy="false">[</mo><mo stretchy="false">]</mo></math>$\text{next}[]$, where next[i]<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>next</mtext><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo></math>$\text{next}[i]$ is the index of the vertex that we reach by following the character i<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi></math>$i$, or 1<math xmlns="http://www.w3.org/1998/Math/MathML"><mo></mo><mn>1</mn></math>$-1$ if there is no such edge. Initially, the trie consists of only one vertex - the root - with the index 0<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>0</mn></math>$0$.

    Now we implement a function that will add a string s<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi></math>$s$ to the trie. The implementation is simple: we start at the root node, and as long as there are edges corresponding to the characters of s<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi></math>$s$ we follow them. If there is no edge for one character, we generate a new vertex and connect it with an edge. At the end of the process we mark the last vertex with the flag output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$.

    void add_string(string const& s) {
    int v = 0;
    for (char ch : s) {
        int c = ch - 'a';
        if (trie[v].next[c] == -1) {
            trie[v].next[c] = trie.size();
            trie.emplace_back();
        }
        v = trie[v].next[c];
    }
    trie[v].output = true;
}

    This implementation obviously runs in linear time, and since every vertex stores k<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>k</mi></math>$k$ links, it will use O(mk)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>m</mi><mi>k</mi><mo stretchy="false">)</mo></math>$O(m k)$ memory.

    It is possible to decrease the memory consumption to O(m)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>m</mi><mo stretchy="false">)</mo></math>$O(m)$ by using a map instead of an array in each vertex. However, this will increase the time complexity to O(mlogk)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>m</mi><mi>log</mi><mo data-mjx-texclass="NONE"></mo><mi>k</mi><mo stretchy="false">)</mo></math>$O(m \log k)$.

    Construction of an automaton

    Suppose we have built a trie for the given set of strings. Now let's look at it from a different side. If we look at any vertex, the string that corresponds to it is a prefix of one or more strings in the set, thus each vertex of the trie can be interpreted as a position in one or more strings from the set.

    In fact, the trie vertices can be interpreted as states in a finite deterministic automaton. From any state we can transition - using some input letter - to other states, i.e., to another position in the set of strings. For example, if there is only one string abc<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi><mi>b</mi><mi>c</mi></math>$abc$ in the dictionary, and we are standing at vertex ab<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi><mi>b</mi></math>$ab$, then using the letter c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$ we can go to the vertex abc<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi><mi>b</mi><mi>c</mi></math>$abc$.

    Thus we can understand the edges of the trie as transitions in an automaton according to the corresponding letter. However, in an automaton we need to have transitions for each combination of a state and a letter. If we try to perform a transition using a letter, and there is no corresponding edge in the trie, then we nevertheless must go into some state.

    More precisely, suppose we are in a state corresponding to a string t<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>t</mi></math>$t$, and we want to transition to a different state using the character c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$. If there is an edge labeled with this letter c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$, then we can simply go over this edge, and get the vertex corresponding to t+c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>t</mi><mo>+</mo><mi>c</mi></math>$t + c$. If there is no such edge, since we want to maintain the invariant that the current state is the longest partial match in the processed string, we must find the longest string in the trie that's a proper suffix of the string t<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>t</mi></math>$t$, and try to perform a transition from there.

    For example, let the trie be constructed by the strings ab<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi><mi>b</mi></math>$ab$ and bc<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>b</mi><mi>c</mi></math>$bc$, and we are currently at the vertex corresponding to ab<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi><mi>b</mi></math>$ab$, which is also an output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertex. To transition with the letter c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$, we are forced to go to the state corresponding to the string b<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>b</mi></math>$b$, and from there follow the edge with the letter c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$.


    An Aho-Corasick automaton based on words "a", "ab", "bc", "bca", "c" and "caa".
    Blue arrows are suffix links, green arrows are terminal links.

    A suffix link for a vertex p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$ is an edge that points to the longest proper suffix of the string corresponding to the vertex p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$. The only special case is the root of the trie, whose suffix link will point to itself. Now we can reformulate the statement about the transitions in the automaton like this: while there is no transition from the current vertex of the trie using the current letter (or until we reach the root), we follow the suffix link.

    Thus we reduced the problem of constructing an automaton to the problem of finding suffix links for all vertices of the trie. However, we will build these suffix links, oddly enough, using the transitions constructed in the automaton.

    The suffix links of the root vertex and all its immediate children point to the root vertex. For any vertex v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ deeper in the tree, we can calculate the suffix link as follows: if p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$ is the ancestor of v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ with c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$ being the letter labeling the edge from p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$ to v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$, go to p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$, then follow its suffix link, and perform the transition with the letter c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$ from there.

    Thus, the problem of finding the transitions has been reduced to the problem of finding suffix links, and the problem of finding suffix links has been reduced to the problem of finding a suffix link and a transition, except for vertices closer to the root. So we have a recursive dependence that we can resolve in linear time.

    Let's move to the implementation. Note that we now will store the ancestor p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$ and the character pch<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mi>c</mi><mi>h</mi></math>$pch$ of the edge from p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$ to v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ for each vertex v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$. Also, at each vertex we will store the suffix link link<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>link</mtext></math>$\text{link}$ (or 1<math xmlns="http://www.w3.org/1998/Math/MathML"><mo></mo><mn>1</mn></math>$-1$ if it hasn't been calculated yet), and in the array go[k]<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>go</mtext><mo stretchy="false">[</mo><mi>k</mi><mo stretchy="false">]</mo></math>$\text{go}[k]$ the transitions in the machine for each symbol (again 1<math xmlns="http://www.w3.org/1998/Math/MathML"><mo></mo><mn>1</mn></math>$-1$ if it hasn't been calculated yet).

    const int K = 26;

struct Vertex {
    int next[K];
    bool output = false;
    int p = -1;
    char pch;
    int link = -1;
    int go[K];

    Vertex(int p=-1, char ch='$') : p(p), pch(ch) {
        fill(begin(next), end(next), -1);
        fill(begin(go), end(go), -1);
    }
};

vector<Vertex> t(1);

void add_string(string const& s) {
    int v = 0;
    for (char ch : s) {
        int c = ch - 'a';
        if (t[v].next[c] == -1) {
            t[v].next[c] = t.size();
            t.emplace_back(v, ch);
        }
        v = t[v].next[c];
    }
    t[v].output = true;
}

int go(int v, char ch);

int get_link(int v) {
    if (t[v].link == -1) {
        if (v == 0 || t[v].p == 0)
            t[v].link = 0;
        else
            t[v].link = go(get_link(t[v].p), t[v].pch);
    }
    return t[v].link;
}

int go(int v, char ch) {
    int c = ch - 'a';
    if (t[v].go[c] == -1) {
        if (t[v].next[c] != -1)
            t[v].go[c] = t[v].next[c];
        else
            t[v].go[c] = v == 0 ? 0 : go(get_link(v), ch);
    }
    return t[v].go[c];
}

    It is easy to see that thanks to memoization of the suffix links and transitions, the total time for finding all suffix links and transitions will be linear.

    For an illustration of the concept refer to slide number 103 of the Stanford slides.

    BFS-based construction

    Instead of computing transitions and suffix links with recursive calls to go and get_link, it is possible to compute them bottom-up starting from the root. (In fact, when the dictionary consists of only one string, we obtain the familiar Knuth-Morris-Pratt algorithm.)

    This approach will have some advantages over the one described above as, instead of the total length m<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi></math>$m$, its running time depends only on the number of vertices n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ in the trie. Moreover, it is possible to adapt it for large alphabets using a persistent array data structure, thus making the construction time O(nlogk)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>n</mi><mi>log</mi><mo data-mjx-texclass="NONE"></mo><mi>k</mi><mo stretchy="false">)</mo></math>$O(n \log k)$ instead of O(mk)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>m</mi><mi>k</mi><mo stretchy="false">)</mo></math>$O(mk)$, which is a significant improvement granted that m<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi></math>$m$ may go up to n2<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>n</mi><mn>2</mn></msup></math>$n^2$.

    We can reason inductively using the fact that BFS from the root traverses vertices in order of increasing length. We may assume that when we're in a vertex v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$, its suffix link u=link[v]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo>=</mo><mi>l</mi><mi>i</mi><mi>n</mi><mi>k</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo></math>$u = link[v]$ is already successfully computed, and for all vertices with shorter length transitions from them are also fully computed.

    Assume that at the moment we stand in a vertex v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ and consider a character c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$. We essentially have two cases:

    1. go[v][c]=1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo><mo>=</mo><mo></mo><mn>1</mn></math>$go[v][c] = -1$. In this case, we may assign go[v][c]=go[u][c]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo><mo>=</mo><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>u</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo></math>$go[v][c] = go[u][c]$, which is already known by the induction hypothesis;
    2. go[v][c]=w1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo><mo>=</mo><mi>w</mi><mo></mo><mo></mo><mn>1</mn></math>$go[v][c] = w \neq -1$. In this case, we may assign link[w]=go[u][c]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>l</mi><mi>i</mi><mi>n</mi><mi>k</mi><mo stretchy="false">[</mo><mi>w</mi><mo stretchy="false">]</mo><mo>=</mo><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>u</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo></math>$link[w] = go[u][c]$.

    In this way, we spend O(1)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mn>1</mn><mo stretchy="false">)</mo></math>$O(1)$ time per each pair of a vertex and a character, making the running time O(mk)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>m</mi><mi>k</mi><mo stretchy="false">)</mo></math>$O(mk)$. The major overhead here is that we copy a lot of transitions from u<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi></math>$u$ in the first case, while the transitions of the second case form the trie and sum up to m<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi></math>$m$ over all vertices. To avoid the copying of go[u][c]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>u</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo></math>$go[u][c]$, we may use a persistent array data structure, using which we initially copy go[u]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>u</mi><mo stretchy="false">]</mo></math>$go[u]$ into go[v]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mi>o</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo></math>$go[v]$ and then only update values for characters in which the transition would differ. This leads to the O(mlogk)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>m</mi><mi>log</mi><mo data-mjx-texclass="NONE"></mo><mi>k</mi><mo stretchy="false">)</mo></math>$O(m \log k)$ algorithm.

    Applications

    Find all strings from a given set in a text

    We are given a set of strings and a text. We have to print all occurrences of all strings from the set in the given text in O(len+ans)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mtext>len</mtext><mo>+</mo><mtext>ans</mtext><mo stretchy="false">)</mo></math>$O(\text{len} + \text{ans})$, where len<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>len</mtext></math>$\text{len}$ is the length of the text and ans<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>ans</mtext></math>$\text{ans}$ is the size of the answer.

    We construct an automaton for this set of strings. We will now process the text letter by letter using the automaton, starting at the root of the trie. If we are at any time at state v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$, and the next letter is c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$, then we transition to the next state with go(v,c)<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>go</mtext><mo stretchy="false">(</mo><mi>v</mi><mo>,</mo><mi>c</mi><mo stretchy="false">)</mo></math>$\text{go}(v, c)$, thereby either increasing the length of the current match substring by 1<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>1</mn></math>$1$, or decreasing it by following a suffix link.

    How can we find out for a state v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$, if there are any matches with strings for the set? First, it is clear that if we stand on a output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertex, then the string corresponding to the vertex ends at this position in the text. However this is by no means the only possible case of achieving a match: if we can reach one or more output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertices by moving along the suffix links, then there will be also a match corresponding to each found output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertex. A simple example demonstrating this situation can be created using the set of strings {dabce,abc,bc}<math xmlns="http://www.w3.org/1998/Math/MathML"><mo fence="false" stretchy="false">{</mo><mi>d</mi><mi>a</mi><mi>b</mi><mi>c</mi><mi>e</mi><mo>,</mo><mi>a</mi><mi>b</mi><mi>c</mi><mo>,</mo><mi>b</mi><mi>c</mi><mo fence="false" stretchy="false">}</mo></math>$\{dabce, abc, bc\}$ and the text dabc<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi><mi>a</mi><mi>b</mi><mi>c</mi></math>$dabc$.

    Thus if we store in each output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertex the index of the string corresponding to it (or the list of indices if duplicate strings appear in the set), then we can find in O(n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>n</mi><mo stretchy="false">)</mo></math>$O(n)$ time the indices of all strings which match the current state, by simply following the suffix links from the current vertex to the root. This is not the most efficient solution, since this results in O(n len)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>n</mi><mtext>&nbsp;</mtext><mtext>len</mtext><mo stretchy="false">)</mo></math>$O(n ~ \text{len})$ complexity overall. However, this can be optimized by computing and storing the nearest output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertex that is reachable using suffix links (this is sometimes called the exit link). This value we can compute lazily in linear time. Thus for each vertex we can advance in O(1)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mn>1</mn><mo stretchy="false">)</mo></math>$O(1)$ time to the next marked vertex in the suffix link path, i.e. to the next match. Thus for each match we spend O(1)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mn>1</mn><mo stretchy="false">)</mo></math>$O(1)$ time, and therefore we reach the complexity O(len+ans)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mtext>len</mtext><mo>+</mo><mtext>ans</mtext><mo stretchy="false">)</mo></math>$O(\text{len} + \text{ans})$.

    If you only want to count the occurrences and not find the indices themselves, you can calculate the number of marked vertices in the suffix link path for each vertex v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$. This can be calculated in O(n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>n</mi><mo stretchy="false">)</mo></math>$O(n)$ time in total. Thus we can sum up all matches in O(len)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mtext>len</mtext><mo stretchy="false">)</mo></math>$O(\text{len})$.

    Finding the lexicographically smallest string of a given length that doesn't match any given strings

    A set of strings and a length L<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>L</mi></math>$L$ is given. We have to find a string of length L<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>L</mi></math>$L$, which does not contain any of the strings, and derive the lexicographically smallest of such strings.

    We can construct the automaton for the set of strings. Recall that output<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>output</mtext></math>$\text{output}$ vertices are the states where we have a match with a string from the set. Since in this task we have to avoid matches, we are not allowed to enter such states. On the other hand we can enter all other vertices. Thus we delete all "bad" vertices from the machine, and in the remaining graph of the automaton we find the lexicographically smallest path of length L<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>L</mi></math>$L$. This task can be solved in O(L)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>L</mi><mo stretchy="false">)</mo></math>$O(L)$ for example by depth first search.

    Finding the shortest string containing all given strings

    Here we use the same ideas. For each vertex we store a mask that denotes the strings which match at this state. Then the problem can be reformulated as follows: initially being in the state (v=root, mask=0)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>v</mi><mo>=</mo><mtext>root</mtext><mo>,</mo><mtext>&nbsp;</mtext><mtext>mask</mtext><mo>=</mo><mn>0</mn><mo stretchy="false">)</mo></math>$(v = \text{root},~ \text{mask} = 0)$, we want to reach the state (v, mask=2n1)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>v</mi><mo>,</mo><mtext>&nbsp;</mtext><mtext>mask</mtext><mo>=</mo><msup><mn>2</mn><mi>n</mi></msup><mo></mo><mn>1</mn><mo stretchy="false">)</mo></math>$(v,~ \text{mask} = 2^n - 1)$, where n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ is the number of strings in the set. When we transition from one state to another using a letter, we update the mask accordingly. By running a breadth first search we can find a path to the state (v, mask=2n1)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>v</mi><mo>,</mo><mtext>&nbsp;</mtext><mtext>mask</mtext><mo>=</mo><msup><mn>2</mn><mi>n</mi></msup><mo></mo><mn>1</mn><mo stretchy="false">)</mo></math>$(v,~ \text{mask} = 2^n - 1)$ with the smallest length.

    Finding the lexicographically smallest string of length L<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>L</mi></math>$L$ containing k<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>k</mi></math>$k$ strings

    As in the previous problem, we calculate for each vertex the number of matches that correspond to it (that is the number of marked vertices reachable using suffix links). We reformulate the problem: the current state is determined by a triple of numbers (v, len, cnt)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>v</mi><mo>,</mo><mtext>&nbsp;</mtext><mtext>len</mtext><mo>,</mo><mtext>&nbsp;</mtext><mtext>cnt</mtext><mo stretchy="false">)</mo></math>$(v,~ \text{len},~ \text{cnt})$, and we want to reach from the state (root, 0, 0)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mtext>root</mtext><mo>,</mo><mtext>&nbsp;</mtext><mn>0</mn><mo>,</mo><mtext>&nbsp;</mtext><mn>0</mn><mo stretchy="false">)</mo></math>$(\text{root},~ 0,~ 0)$ the state (v, L, k)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>v</mi><mo>,</mo><mtext>&nbsp;</mtext><mi>L</mi><mo>,</mo><mtext>&nbsp;</mtext><mi>k</mi><mo stretchy="false">)</mo></math>$(v,~ L,~ k)$, where v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ can be any vertex. Thus we can find such a path using depth first search (and if the search looks at the edges in their natural order, then the found path will automatically be the lexicographically smallest).
