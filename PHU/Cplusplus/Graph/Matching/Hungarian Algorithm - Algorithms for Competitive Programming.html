    Hungarian algorithm for solving the assignment problem

    Statement of the assignment problem

    There are several standard formulations of the assignment problem (all of which are essentially equivalent). Here are some of them:

    • There are n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ jobs and n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ workers. Each worker specifies the amount of money they expect for a particular job. Each worker can be assigned to only one job. The objective is to assign jobs to workers in a way that minimizes the total cost.

    • Given an nn<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo></mo><mi>n</mi></math>$n \times n$ matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$, the task is to select one number from each row such that exactly one number is chosen from each column, and the sum of the selected numbers is minimized.

    • Given an nn<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo></mo><mi>n</mi></math>$n \times n$ matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$, the task is to find a permutation p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$ of length n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ such that the value A[i][p[i]]<math xmlns="http://www.w3.org/1998/Math/MathML"><mo data-mjx-texclass="OP"></mo><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mrow data-mjx-texclass="INNER"><mo data-mjx-texclass="OPEN">[</mo><mi>p</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo data-mjx-texclass="CLOSE">]</mo></mrow></math>$\sum A[i]\left[p[i]\right]$ is minimized.

    • Consider a complete bipartite graph with n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ vertices per part, where each edge is assigned a weight. The objective is to find a perfect matching with the minimum total weight.

    It is important to note that all the above scenarios are "square" problems, meaning both dimensions are always equal to n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$. In practice, similar "rectangular" formulations are often encountered, where n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ is not equal to m<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi></math>$m$, and the task is to select min(n,m)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo data-mjx-texclass="OP" movablelimits="true">min</mo><mo stretchy="false">(</mo><mi>n</mi><mo>,</mo><mi>m</mi><mo stretchy="false">)</mo></math>$\min(n,m)$ elements. However, it can be observed that a "rectangular" problem can always be transformed into a "square" problem by adding rows or columns with zero or infinite values, respectively.

    We also note that by analogy with the search for a minimum solution, one can also pose the problem of finding a maximum solution. However, these two problems are equivalent to each other: it is enough to multiply all the weights by 1<math xmlns="http://www.w3.org/1998/Math/MathML"><mo></mo><mn>1</mn></math>$-1$.

    Hungarian algorithm

    Historical reference

    The algorithm was developed and published by Harold Kuhn in 1955. Kuhn himself gave it the name "Hungarian" because it was based on the earlier work by Hungarian mathematicians Dnes Knig and Jen Egervry.
    In 1957, James Munkres showed that this algorithm runs in (strictly) polynomial time, independently from the cost.
    Therefore, in literature, this algorithm is known not only as the "Hungarian", but also as the "Kuhn-Mankres algorithm" or "Mankres algorithm".
    However, it was recently discovered in 2006 that the same algorithm was invented a century before Kuhn by the German mathematician Carl Gustav Jacobi. His work, About the research of the order of a system of arbitrary ordinary differential equations, which was published posthumously in 1890, contained, among other findings, a polynomial algorithm for solving the assignment problem. Unfortunately, since the publication was in Latin, it went unnoticed among mathematicians.

    It is also worth noting that Kuhn's original algorithm had an asymptotic complexity of O(n4)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><msup><mi>n</mi><mn>4</mn></msup><mo stretchy="false">)</mo></math>$\mathcal{O}(n^4)$, and only later Jack Edmonds and Richard Karp (and independently Tomizawa) showed how to improve it to an asymptotic complexity of O(n3)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><msup><mi>n</mi><mn>3</mn></msup><mo stretchy="false">)</mo></math>$\mathcal{O}(n^3)$.

    The O(n4)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><msup><mi>n</mi><mn>4</mn></msup><mo stretchy="false">)</mo></math>$\mathcal{O}(n^4)$ algorithm

    To avoid ambiguity, we note right away that we are mainly concerned with the assignment problem in a matrix formulation (i.e., given a matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$, you need to select n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ cells from it that are in different rows and columns). We index arrays starting with 1<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>1</mn></math>$1$, i.e., for example, a matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$ has indices A[1n][1n]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo></math>$A[1 \dots n][1 \dots n]$.

    We will also assume that all numbers in matrix A are non-negative (if this is not the case, you can always make the matrix non-negative by adding some constant to all numbers).

    Let's call a potential two arbitrary arrays of numbers u[1n]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo></math>$u[1 \ldots n]$ and v[1n]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo></math>$v[1 \ldots n]$, such that the following condition is satisfied:

    u[i]+v[j]A[i][j],i=1n, j=1n<math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo>+</mo><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo></mo><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>,</mo><mstyle scriptlevel="0"><mspace width="1em"></mspace></mstyle><mi>i</mi><mo>=</mo><mn>1</mn><mo></mo><mi>n</mi><mo>,</mo><mtext>&nbsp;</mtext><mi>j</mi><mo>=</mo><mn>1</mn><mo></mo><mi>n</mi></math>$$u[i]+v[j]\leq A[i][j],\quad i=1\dots n,\ j=1\dots n$$

    (As you can see, u[i]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo></math>$u[i]$ corresponds to the i<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi></math>$i$-th row, and v[j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$v[j]$ corresponds to the j<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi></math>$j$-th column of the matrix).

    Let's call the value f<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi></math>$f$ of the potential the sum of its elements:

    f=i=1nu[i]+j=1nv[j].<math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><mi>f</mi><mo>=</mo><munderover><mo data-mjx-texclass="OP"></mo><mrow data-mjx-texclass="ORD"><mi>i</mi><mo>=</mo><mn>1</mn></mrow><mrow data-mjx-texclass="ORD"><mi>n</mi></mrow></munderover><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo>+</mo><munderover><mo data-mjx-texclass="OP"></mo><mrow data-mjx-texclass="ORD"><mi>j</mi><mo>=</mo><mn>1</mn></mrow><mrow data-mjx-texclass="ORD"><mi>n</mi></mrow></munderover><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>.</mo></math>$$f=\sum_{i=1}^{n} u[i] + \sum_{j=1}^{n} v[j].$$

    On one hand, it is easy to see that the cost of the desired solution sol<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi><mi>o</mi><mi>l</mi></math>$sol$ is not less than the value of any potential.

    Lemma. solf.<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi><mi>o</mi><mi>l</mi><mo></mo><mi>f</mi><mo>.</mo></math>$sol\geq f.$

    Proof

    The desired solution of the problem consists of n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ cells of the matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$, so u[i]+v[j]A[i][j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo>+</mo><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo></mo><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$u[i]+v[j]\leq A[i][j]$ for each of them. Since all the elements in sol<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi><mi>o</mi><mi>l</mi></math>$sol$ are in different rows and columns, summing these inequalities over all the selected A[i][j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$A[i][j]$, you get f<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi></math>$f$ on the left side of the inequality, and sol<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi><mi>o</mi><mi>l</mi></math>$sol$ on the right side.

    On the other hand, it turns out that there is always a solution and a potential that turns this inequality into equality. The Hungarian algorithm described below will be a constructive proof of this fact. For now, let's just pay attention to the fact that if any solution has a cost equal to any potential, then this solution is optimal.

    Let's fix some potential. Let's call an edge (i,j)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>i</mi><mo>,</mo><mi>j</mi><mo stretchy="false">)</mo></math>$(i,j)$ rigid if u[i]+v[j]=A[i][j].<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo>+</mo><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>=</mo><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>.</mo></math>$u[i]+v[j]=A[i][j].$

    Recall an alternative formulation of the assignment problem, using a bipartite graph. Denote with H<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>H</mi></math>$H$ a bipartite graph composed only of rigid edges. The Hungarian algorithm will maintain, for the current potential, the maximum-number-of-edges matching M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ of the graph H<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>H</mi></math>$H$. As soon as M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ contains n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ edges, then the solution to the problem will be just M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ (after all, it will be a solution whose cost coincides with the value of a potential).

    Let's proceed directly to the description of the algorithm.

    Step 1. At the beginning, the potential is assumed to be zero (u[i]=v[i]=0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo>=</mo><mi>v</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo>=</mo><mn>0</mn></math>$u[i]=v[i]=0$ for all i<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi></math>$i$), and the matching M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ is assumed to be empty.

    Step 2. Further, at each step of the algorithm, we try, without changing the potential, to increase the cardinality of the current matching M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ by one (recall that the matching is searched in the graph of rigid edges H<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>H</mi></math>$H$). To do this, the usual Kuhn Algorithm for finding the maximum matching in bipartite graphs is used. Let us recall the algorithm here. All edges of the matching M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ are oriented in the direction from the right part to the left one, and all other edges of the graph H<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>H</mi></math>$H$ are oriented in the opposite direction.

    Recall (from the terminology of searching for matchings) that a vertex is called saturated if an edge of the current matching is adjacent to it. A vertex that is not adjacent to any edge of the current matching is called unsaturated. A path of odd length, in which the first edge does not belong to the matching, and for all subsequent edges there is an alternating belonging to the matching (belongs/does not belong) - is called an augmenting path. From all unsaturated vertices in the left part, a depth-first or breadth-first traversal is started. If, as a result of the search, it was possible to reach an unsaturated vertex of the right part, we have found an augmenting path from the left part to the right one. If we include odd edges of the path and remove the even ones in the matching (i.e. include the first edge in the matching, exclude the second, include the third, etc.), then we will increase the matching cardinality by one.

    If there was no augmenting path, then the current matching M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ is maximal in the graph H<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>H</mi></math>$H$.

    Step 3. If at the current step, it is not possible to increase the cardinality of the current matching, then a recalculation of the potential is performed in such a way that, at the next steps, there will be more opportunities to increase the matching.

    Denote by Z1<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>Z</mi><mn>1</mn></msub></math>$Z_1$ the set of vertices of the left part that were visited during the last traversal of Kuhn's algorithm, and through Z2<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>Z</mi><mn>2</mn></msub></math>$Z_2$ the set of visited vertices of the right part.

    Let's calculate the value <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$:

    =miniZ1, jZ2A[i][j]u[i]v[j].<math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><mi mathvariant="normal"></mi><mo>=</mo><munder><mo data-mjx-texclass="OP" movablelimits="true">min</mo><mrow data-mjx-texclass="ORD"><mi>i</mi><mo></mo><msub><mi>Z</mi><mn>1</mn></msub><mo>,</mo><mtext>&nbsp;</mtext><mi>j</mi><mo></mo><msub><mi>Z</mi><mn>2</mn></msub></mrow></munder><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo></mo><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo></mo><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>.</mo></math>$$\Delta = \min_{i\in Z_1,\ j\notin Z_2} A[i][j]-u[i]-v[j].$$

    Lemma. >0.<math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi><mo>&gt;</mo><mn>0.</mn></math>$\Delta > 0.$

    Proof

    Suppose =0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi><mo>=</mo><mn>0</mn></math>$\Delta=0$. Then there exists a rigid edge (i,j)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>i</mi><mo>,</mo><mi>j</mi><mo stretchy="false">)</mo></math>$(i,j)$ with iZ1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi><mo></mo><msub><mi>Z</mi><mn>1</mn></msub></math>$i\in Z_1$ and jZ2<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi><mo></mo><msub><mi>Z</mi><mn>2</mn></msub></math>$j\notin Z_2$.     Now let's learn how to implement the same algorithm in O(n3)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><msup><mi>n</mi><mn>3</mn></msup><mo stretchy="false">)</mo></math>$\mathcal{O}(n^3)$ (for rectangular problems nm<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo></mo><mi>m</mi></math>$n \times m$, O(n2m)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><msup><mi>n</mi><mn>2</mn></msup><mi>m</mi><mo stretchy="false">)</mo></math>$\mathcal{O}(n^2m)$).

    The key idea is to consider matrix rows one by one, and not all at once. Thus, the algorithm described above will take the following form:

    1. Consider the next row of the matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$.

    2. While there is no increasing path starting in this row, recalculate the potential.

    3. As soon as an augmenting path is found, propagate the matching along it (thus including the last edge in the matching), and restart from step 1 (to consider the next line).

    To achieve the required complexity, it is necessary to implement steps 2-3, which are performed for each row of the matrix, in time O(n2)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><msup><mi>n</mi><mn>2</mn></msup><mo stretchy="false">)</mo></math>$\mathcal{O}(n^2)$ (for rectangular problems in O(nm)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><mi>n</mi><mi>m</mi><mo stretchy="false">)</mo></math>$\mathcal{O}(nm)$).

    To do this, recall two facts proved above:

    • With a change in the potential, the vertices that were reachable by Kuhn's traversal will remain reachable.

    • In total, only O(n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><mi>n</mi><mo stretchy="false">)</mo></math>$\mathcal{O}(n)$ recalculations of the potential could occur before an augmenting path was found.

    From this follow these key ideas that allow us to achieve the required complexity:

    • To check for the presence of an augmenting path, there is no need to start the Kuhn traversal again after each potential recalculation. Instead, you can make the Kuhn traversal in an iterative form: after each recalculation of the potential, look at the added rigid edges and, if their left ends were reachable, mark their right ends reachable as well and continue the traversal from them.

    • Developing this idea further, we can present the algorithm as follows: at each step of the loop, the potential is recalculated. Subsequently, a column that has become reachable is identified (which will always exist as new reachable vertices emerge after every potential recalculation). If the column is unsaturated, an augmenting chain is discovered. Conversely, if the column is saturated, the matching row also becomes reachable.

    • To quickly recalculate the potential (faster than the O(n2)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><msup><mi>n</mi><mn>2</mn></msup><mo stretchy="false">)</mo></math>$\mathcal{O}(n^2)$ naive version), you need to maintain auxiliary minima for each of the columns:


      minv[j]=miniZ1A[i][j]u[i]v[j].<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi><mi>i</mi><mi>n</mi><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>=</mo><munder><mo data-mjx-texclass="OP" movablelimits="true">min</mo><mrow data-mjx-texclass="ORD"><mi>i</mi><mo></mo><msub><mi>Z</mi><mn>1</mn></msub></mrow></munder><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo></mo><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo></mo><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>.</mo></math>$minv[j]=\min_{i\in Z_1} A[i][j]-u[i]-v[j].$

      It's easy to see that the desired value <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$ is expressed in terms of them as follows:


      =minjZ2minv[j].<math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi><mo>=</mo><munder><mo data-mjx-texclass="OP" movablelimits="true">min</mo><mrow data-mjx-texclass="ORD"><mi>j</mi><mo></mo><msub><mi>Z</mi><mn>2</mn></msub></mrow></munder><mi>m</mi><mi>i</mi><mi>n</mi><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo>.</mo></math>$\Delta=\min_{j\notin Z_2} minv[j].$

      Thus, finding <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$ can now be done in O(n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><mi>n</mi><mo stretchy="false">)</mo></math>$\mathcal{O}(n)$.

      It is necessary to update the array minv<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi><mi>i</mi><mi>n</mi><mi>v</mi></math>$minv$ when new visited rows appear.     Thus, the algorithm takes the following form: in the outer loop, we consider matrix rows one by one.     Implementation of the Hungarian algorithm

    The implementation below was developed by Andrey Lopatin several years ago. It is distinguished by amazing conciseness: the entire algorithm consists of 30 lines of code.

    The implementation finds a solution for the rectangular matrix A[1n][1m]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>m</mi><mo stretchy="false">]</mo></math>$A[1\dots n][1\dots m]$, where nm<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo></mo><mi>m</mi></math>$n\leq m$. The matrix is 1-based for convenience and code brevity: this implementation introduces a dummy zero row and zero column, which allows us to write many cycles in a general form, without additional checks.

    Arrays u[0n]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mn>0</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo></math>$u[0 \ldots n]$ and v[0m]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi><mo stretchy="false">[</mo><mn>0</mn><mo></mo><mi>m</mi><mo stretchy="false">]</mo></math>$v[0 \ldots m]$ store potential. Initially, they are set to zero, which is consistent with a matrix of zero rows (Note that it is unimportant for this implementation whether or not the matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$ contains negative numbers).

    The array p[0m]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo stretchy="false">[</mo><mn>0</mn><mo></mo><mi>m</mi><mo stretchy="false">]</mo></math>$p[0 \ldots m]$ contains a matching: for each column j=1m<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi><mo>=</mo><mn>1</mn><mo></mo><mi>m</mi></math>$j = 1 \ldots m$, it stores the number p[j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$p[j]$ of the selected row (or 0<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>0</mn></math>$0$ if nothing has been selected yet). For the convenience of implementation, p[0]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo stretchy="false">[</mo><mn>0</mn><mo stretchy="false">]</mo></math>$p[0]$ is assumed to be equal to the number of the current row.

    The array minv[1m]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi><mi>i</mi><mi>n</mi><mi>v</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>m</mi><mo stretchy="false">]</mo></math>$minv[1 \ldots m]$ contains, for each column j<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi></math>$j$, the auxiliary minima necessary for a quick recalculation of the potential, as described above.

    The array way[1m]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>w</mi><mi>a</mi><mi>y</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>m</mi><mo stretchy="false">]</mo></math>$way[1 \ldots m]$ contains information about where these minimums are reached so that we can later reconstruct the augmenting path. Note that, to reconstruct the path, it is sufficient to store only column values, since the row numbers can be taken from the matching (i.e., from the array p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$). Thus, way[j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>w</mi><mi>a</mi><mi>y</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$way[j]$, for each column j<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi></math>$j$, contains the number of the previous column in the path (or 0<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>0</mn></math>$0$ if there is none).

    The algorithm itself is an outer loop through the rows of the matrix, inside which the i<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi></math>$i$-th row of the matrix is considered. The first do-while loop runs until a free column j0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi><mn>0</mn></math>$j0$ is found. Each iteration of the loop marks visited a new column with the number j0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi><mn>0</mn></math>$j0$ (calculated at the last iteration; and initially equal to zero - i.e. we start from a dummy column), as well as a new row i0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi><mn>0</mn></math>$i0$ - adjacent to it in the matching (i.e. p[j0]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo stretchy="false">[</mo><mi>j</mi><mn>0</mn><mo stretchy="false">]</mo></math>$p[j0]$; and initially when j0=0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi><mn>0</mn><mo>=</mo><mn>0</mn></math>$j0=0$ the i<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi></math>$i$-th row is taken). Due to the appearance of a new visited row i0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi><mn>0</mn></math>$i0$, you need to recalculate the array minv<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi><mi>i</mi><mi>n</mi><mi>v</mi></math>$minv$ and <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$ accordingly. If <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$ is updated, then the column j1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi><mn>1</mn></math>$j1$ becomes the minimum that has been reached (note that with such an implementation <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$ could turn out to be equal to zero, which means that the potential cannot be changed at the current step: there is already a new reachable column). After that, the potential and the minv<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>m</mi><mi>i</mi><mi>n</mi><mi>v</mi></math>$minv$ array are recalculated. At the end of the "do-while" loop, we found an augmenting path ending in a column j0<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi><mn>0</mn></math>$j0$ that can be "unrolled" using the ancestor array way<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>w</mi><mi>a</mi><mi>y</mi></math>$way$.

    The constant INF is "infinity", i.e. some number, obviously greater than all possible numbers in the input matrix A<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi></math>$A$.

    vector<int> u (n+1), v (m+1), p (m+1), way (m+1);
for (int i=1; i<=n; ++i) {
    p[0] = i;
    int j0 = 0;
    vector<int> minv (m+1, INF);
    vector<bool> used (m+1, false);
    do {
        used[j0] = true;
        int i0 = p[j0],  delta = INF,  j1;
        for (int j=1; j<=m; ++j)
            if (!used[j]) {
                int cur = A[i0][j]-u[i0]-v[j];
                if (cur < minv[j])
                    minv[j] = cur,  way[j] = j0;
                if (minv[j] < delta)
                    delta = minv[j],  j1 = j;
            }
        for (int j=0; j<=m; ++j)
            if (used[j])
                u[p[j]] += delta,  v[j] -= delta;
            else
                minv[j] -= delta;
        j0 = j1;
    } while (p[j0] != 0);
    do {
        int j1 = way[j0];
        p[j0] = p[j1];
        j0 = j1;
    } while (j0);
}

Indeed, as you can see from the code, v[0]<math xmlns="http://www.w3.org/1998/Math/MathML"><mo></mo><mi>v</mi><mo stretchy="false">[</mo><mn>0</mn><mo stretchy="false">]</mo></math>$-v[0]$ contains the sum of all the values of <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$, i.e. total change in potential. Although several values of u[i]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo></math>$u[i]$ and v[j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$v[j]$ could change at once, the total change in the potential is exactly equal to <math xmlns="http://www.w3.org/1998/Math/MathML"><mi mathvariant="normal"></mi></math>$\Delta$, since until there is an augmenting path, the number of reachable rows is exactly one more than the number of the reachable columns (only the current row i<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi></math>$i$ does not have a "pair" in the form of a visited column):

    Connection to the Successive Shortest Path Algorithm

    The Hungarian algorithm can be seen as the Successive Shortest Path Algorithm, adapted for the assignment problem. Without going into the details, let's provide an intuition regarding the connection between them.

    The Successive Path algorithm uses a modified version of Johnson's algorithm as reweighting technique. This one is divided into four steps:

    • Use the Bellman-Ford algorithm, starting from the sink s<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi></math>$s$ and, for each node, find the minimum weight h(v)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>h</mi><mo stretchy="false">(</mo><mi>v</mi><mo stretchy="false">)</mo></math>$h(v)$ of a path from s<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi></math>$s$ to v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$.

    For every step of the main algorithm:

    • Reweight the edges of the original graph in this way: w(u,v)w(u,v)+h(u)h(v)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>w</mi><mo stretchy="false">(</mo><mi>u</mi><mo>,</mo><mi>v</mi><mo stretchy="false">)</mo><mo stretchy="false"></mo><mi>w</mi><mo stretchy="false">(</mo><mi>u</mi><mo>,</mo><mi>v</mi><mo stretchy="false">)</mo><mo>+</mo><mi>h</mi><mo stretchy="false">(</mo><mi>u</mi><mo stretchy="false">)</mo><mo></mo><mi>h</mi><mo stretchy="false">(</mo><mi>v</mi><mo stretchy="false">)</mo></math>$w(u,v) \gets w(u,v)+h(u)-h(v)$.
    • Use Dijkstra's algorithm to find the shortest-paths subgraph of the original network.
    • Update potentials for the next iteration.

    Given this description, we can observe that there is a strong analogy between h(v)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>h</mi><mo stretchy="false">(</mo><mi>v</mi><mo stretchy="false">)</mo></math>$h(v)$ and potentials: it can be checked that they are equal up to a constant offset. In addition, it can be shown that, after reweighting, the set of all zero-weight edges represents the shortest-path subgraph where the main algorithm tries to increase the flow. This also happens in the Hungarian algorithm: we create a subgraph made of rigid edges (the ones for which the quantity A[i][j]u[i]v[j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo></mo><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo></mo><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$A[i][j]-u[i]-v[j]$ is zero), and we try to increase the size of the matching.

    In step 4, all the h(v)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>h</mi><mo stretchy="false">(</mo><mi>v</mi><mo stretchy="false">)</mo></math>$h(v)$ are updated: every time we modify the flow network, we should guarantee that the distances from the source are correct (otherwise, in the next iteration, Dijkstra's algorithm might fail). This sounds like the update performed on the potentials, but in this case, they are not equally incremented.

    Task examples

    Here are a few examples related to the assignment problem, from very trivial to less obvious tasks:

    • Given a bipartite graph, it is required to find in it the maximum matching with the minimum weight (i.e., first of all, the size of the matching is maximized, and secondly, its cost is minimized).
      To solve it, we simply build an assignment problem, putting the number "infinity" in place of the missing edges. After that, we solve the problem with the Hungarian algorithm, and remove edges of infinite weight from the answer (they could enter the answer if the problem does not have a solution in the form of a perfect matching).

    • Given a bipartite graph, it is required to find in it the maximum matching with the maximum weight.
      The solution is again obvious, all weights must be multiplied by minus one.

    • The task of detecting moving objects in images: two images were taken, as a result of which two sets of coordinates were obtained. It is required to correlate the objects in the first and second images, i.e. determine for each point of the second image, which point of the first image it corresponded to. In this case, it is required to minimize the sum of distances between the compared points (i.e., we are looking for a solution in which the objects have taken the shortest path in total).
      To solve, we simply build and solve an assignment problem, where the weights of the edges are the Euclidean distances between points.

    • The task of detecting moving objects by locators: there are two locators that can't determine the position of an object in space, but only its direction. Both locators (located at different points) received information in the form of n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ such directions. It is required to determine the position of objects, i.e. determine the expected positions of objects and their corresponding pairs of directions in such a way that the sum of distances from objects to direction rays is minimized.
      Solution: again, we simply build and solve the assignment problem, where the vertices of the left part are the n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ directions from the first locator, the vertices of the right part are the n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ directions from the second locator, and the weights of the edges are the distances between the corresponding rays.

    • Covering a directed acyclic graph with paths: given a directed acyclic graph, it is required to find the smallest number of paths (if equal, with the smallest total weight) so that each vertex of the graph lies in exactly one path.
      The solution is to build the corresponding bipartite graph from the given graph and find the maximum matching of the minimum weight in it. See separate article for more details.

    • Tree coloring book. Given a tree in which each vertex, except for leaves, has exactly k1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>k</mi><mo></mo><mn>1</mn></math>$k-1$ children. It is required to choose for each vertex one of the k<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>k</mi></math>$k$ colors available so that no two adjacent vertices have the same color. In addition, for each vertex and each color, the cost of painting this vertex with this color is known, and it is required to minimize the total cost.
      To solve this problem, we use dynamic programming. Namely, let's learn how to calculate the value d[v][c]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo></math>$d[v][c]$, where v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ is the vertex number, c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$ is the color number, and the value d[v][c]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo></math>$d[v][c]$ itself is the minimum cost needed to color all the vertices in the subtree rooted at v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$, and the vertex v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ itself with color c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$. To calculate such a value d[v][c]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo></math>$d[v][c]$, it is necessary to distribute the remaining k1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>k</mi><mo></mo><mn>1</mn></math>$k-1$ colors among the children of the vertex v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$, and for this, it is necessary to build and solve the assignment problem (in which the vertices of the left part are colors, the vertices of the right part are children, and the weights of the edges are the corresponding values of d<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi></math>$d$).
      Thus, each value d[v][c]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi><mo stretchy="false">[</mo><mi>v</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>c</mi><mo stretchy="false">]</mo></math>$d[v][c]$ is calculated using the solution of the assignment problem, which ultimately gives the asymptotic O(nk4)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><mi>n</mi><msup><mi>k</mi><mn>4</mn></msup><mo stretchy="false">)</mo></math>$\mathcal{O}(nk^4)$.

    • If, in the assignment problem, the weights are not on the edges, but on the vertices, and only on the vertices of the same part, then it's not necessary to use the Hungarian algorithm: just sort the vertices by weight and run the usual Kuhn algorithm (for more details, see a separate article).

    • Consider the following special case. Let each vertex of the left part be assigned some number [i]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo></math>$\alpha[i]$, and each vertex of the right part [j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$\beta[j]$. Let the weight of any edge (i,j)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">(</mo><mi>i</mi><mo>,</mo><mi>j</mi><mo stretchy="false">)</mo></math>$(i,j)$ be equal to [i][j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo></mo><mi></mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$\alpha[i]\cdot \beta[j]$ (the numbers [i]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo></math>$\alpha[i]$ and [j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$\beta[j]$ are known). Solve the assignment problem.
      To solve it without the Hungarian algorithm, we first consider the case when both parts have two vertices. In this case, as you can easily see, it is better to connect the vertices in the reverse order: connect the vertex with the smaller [i]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo></math>$\alpha[i]$ to the vertex with the larger [j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$\beta[j]$. This rule can be easily generalized to an arbitrary number of vertices: you need to sort the vertices of the first part in increasing order of [i]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo></math>$\alpha[i]$ values, the second part in decreasing order of [j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi></mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$\beta[j]$ values, and connect the vertices in pairs in that order. Thus, we obtain a solution with complexity of O(nlogn)<math xmlns="http://www.w3.org/1998/Math/MathML"><mrow data-mjx-texclass="ORD"><mi data-mjx-variant="-tex-calligraphic" mathvariant="script">O</mi></mrow><mo stretchy="false">(</mo><mi>n</mi><mi>log</mi><mo data-mjx-texclass="NONE"></mo><mi>n</mi><mo stretchy="false">)</mo></math>$\mathcal{O}(n\log n)$.

    • The Problem of Potentials. Given a matrix A[1n][1m]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>A</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>m</mi><mo stretchy="false">]</mo></math>$A[1 \ldots n][1 \ldots m]$, it is required to find two arrays u[1n]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>n</mi><mo stretchy="false">]</mo></math>$u[1 \ldots n]$ and v[1m]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi><mo stretchy="false">[</mo><mn>1</mn><mo></mo><mi>m</mi><mo stretchy="false">]</mo></math>$v[1 \ldots m]$ such that, for any i<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>i</mi></math>$i$ and j<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>j</mi></math>$j$, u[i]+v[j]a[i][j]<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo>+</mo><mi>v</mi><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo><mo></mo><mi>a</mi><mo stretchy="false">[</mo><mi>i</mi><mo stretchy="false">]</mo><mo stretchy="false">[</mo><mi>j</mi><mo stretchy="false">]</mo></math>$u[i] + v[j] \leq a[i][j]$ and the sum of elements of arrays u<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi></math>$u$ and v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>v</mi></math>$v$ is maximum.
      Knowing the Hungarian algorithm, the solution to this problem will not be difficult: the Hungarian algorithm just finds such a potential u,v<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>u</mi><mo>,</mo><mi>v</mi></math>$u, v$ that satisfies the condition of the problem. On the other hand, without knowledge of the Hungarian algorithm, it seems almost impossible to solve such a problem.

      Remark

      This task is also called the dual problem of the assignment problem: minimizing the total cost of the assignment is equivalent to maximizing the sum of the potentials.
