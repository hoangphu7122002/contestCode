    Integer factorization¶

    In this article we list several algorithms for the factorization of integers, each of which can be either fast or varying levels of slow depending on their input.

    Notice, if the number that you want to factorize is actually a prime number, most of the algorithms will run very slowly. This is especially true for Fermat's, Pollard's p-1 and Pollard's rho factorization algorithms. Therefore, it makes the most sense to perform a probabilistic (or a fast deterministic) primality test before trying to factorize the number.

    Trial division¶

    This is the most basic algorithm to find a prime factorization.

    We divide by each possible divisor   d<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi></math>$d$ . It can be observed that it is impossible for all prime factors of a composite number   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$  to be bigger than   n<math xmlns="http://www.w3.org/1998/Math/MathML"><msqrt><mi>n</mi></msqrt></math>$\sqrt{n}$ . Therefore, we only need to test the divisors   2≤d≤n<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>2</mn><mo>≤</mo><mi>d</mi><mo>≤</mo><msqrt><mi>n</mi></msqrt></math>$2 \le d \le \sqrt{n}$ , which gives us the prime factorization in   O(n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><msqrt><mi>n</mi></msqrt><mo stretchy="false">)</mo></math>$O(\sqrt{n})$ . (This is pseudo-polynomial time, i.e. polynomial in the value of the input but exponential in the number of bits of the input.)

    The smallest divisor must be a prime number. We remove the factored number, and continue the process. If we cannot find any divisor in the range   [2;n]<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">[</mo><mn>2</mn><mo>;</mo><msqrt><mi>n</mi></msqrt><mo stretchy="false">]</mo></math>$[2; \sqrt{n}]$ , then the number itself has to be prime.

    vector<long long> trial_division1(long long n) {
    vector<long long> factorization;
    for (long long d = 2; d * d <= n; d++) {
        while (n % d == 0) {
            factorization.push_back(d);
            n /= d;
        }
    }
    if (n > 1)
        factorization.push_back(n);
    return factorization;
}

    Wheel factorization¶

    This is an optimization of the trial division. Once we know that the number is not divisible by 2, we don't need to check other even numbers. This leaves us with only   50%<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>50</mn><mi mathvariant="normal">%</mi></math>$50\%$  of the numbers to check. After factoring out 2, and getting an odd number, we can simply start with 3 and only count other odd numbers.

    vector<long long> trial_division2(long long n) {
    vector<long long> factorization;
    while (n % 2 == 0) {
        factorization.push_back(2);
        n /= 2;
    }
    for (long long d = 3; d * d <= n; d += 2) {
        while (n % d == 0) {
            factorization.push_back(d);
            n /= d;
        }
    }
    if (n > 1)
        factorization.push_back(n);
    return factorization;
}

    This method can be extended further. If the number is not divisible by 3, we can also ignore all other multiples of 3 in the future computations. So we only need to check the numbers   5,7,11,13,17,19,23,…<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>5</mn><mo>,</mo><mn>7</mn><mo>,</mo><mn>11</mn><mo>,</mo><mn>13</mn><mo>,</mo><mn>17</mn><mo>,</mo><mn>19</mn><mo>,</mo><mn>23</mn><mo>,</mo><mo>…</mo></math>$5, 7, 11, 13, 17, 19, 23, \dots$ . We can observe a pattern of these remaining numbers. We need to check all numbers with   dmod6=1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mn>6</mn><mo>=</mo><mn>1</mn></math>$d \bmod 6 = 1$  and   dmod6=5<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>d</mi><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mn>6</mn><mo>=</mo><mn>5</mn></math>$d \bmod 6 = 5$ . So this leaves us with only   33.3%<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>33.3</mn><mi mathvariant="normal">%</mi></math>$33.3\%$  percent of the numbers to check. We can implement this by factoring out the primes 2 and 3 first, after which we start with 5 and only count remainders   1<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>1</mn></math>$1$  and   5<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>5</mn></math>$5$  modulo   6<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>6</mn></math>$6$ .

    Here is an implementation for the prime number 2, 3 and 5. It is convenient to store the skipping strides in an array.

    vector<long long> trial_division3(long long n) {
    vector<long long> factorization;
    for (int d : {2, 3, 5}) {
        while (n % d == 0) {
            factorization.push_back(d);
            n /= d;
        }
    }
    static array<int, 8> increments = {4, 2, 4, 2, 4, 6, 2, 6};
    int i = 0;
    for (long long d = 7; d * d <= n; d += increments[i++]) {
        while (n % d == 0) {
            factorization.push_back(d);
            n /= d;
        }
        if (i == 8)
            i = 0;
    }
    if (n > 1)
        factorization.push_back(n);
    return factorization;
}

    If we continue exending this method to include even more primes, better percentages can be reached, but the skip lists will become larger.

    Precomputed primes¶

    Extending the wheel factorization method indefinitely, we will only be left with prime numbers to check. A good way of checking this is to precompute all prime numbers with the Sieve of Eratosthenes until   n<math xmlns="http://www.w3.org/1998/Math/MathML"><msqrt><mi>n</mi></msqrt></math>$\sqrt{n}$ , and test them individually.

    vector<long long> primes;

vector<long long> trial_division4(long long n) {
    vector<long long> factorization;
    for (long long d : primes) {
        if (d * d > n)
            break;
        while (n % d == 0) {
            factorization.push_back(d);
            n /= d;
        }
    }
    if (n > 1)
        factorization.push_back(n);
    return factorization;
}

    Fermat's factorization method¶

    We can write an odd composite number   n=p⋅q<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo>=</mo><mi>p</mi><mo>⋅</mo><mi>q</mi></math>$n = p \cdot q$  as the difference of two squares   n=a2−b2<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo>=</mo><msup><mi>a</mi><mn>2</mn></msup><mo>−</mo><msup><mi>b</mi><mn>2</mn></msup></math>$n = a^2 - b^2$ :

      n=(p+q2)2−(p−q2)2<math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><mi>n</mi><mo>=</mo><msup><mrow data-mjx-texclass="INNER"><mo data-mjx-texclass="OPEN">(</mo><mfrac><mrow><mi>p</mi><mo>+</mo><mi>q</mi></mrow><mn>2</mn></mfrac><mo data-mjx-texclass="CLOSE">)</mo></mrow><mn>2</mn></msup><mo>−</mo><msup><mrow data-mjx-texclass="INNER"><mo data-mjx-texclass="OPEN">(</mo><mfrac><mrow><mi>p</mi><mo>−</mo><mi>q</mi></mrow><mn>2</mn></mfrac><mo data-mjx-texclass="CLOSE">)</mo></mrow><mn>2</mn></msup></math>$$n = \left(\frac{p + q}{2}\right)^2 - \left(\frac{p - q}{2}\right)^2$$ 

    Fermat's factorization method tries to exploit this fact by guessing the first square   a2<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>a</mi><mn>2</mn></msup></math>$a^2$ , and checking if the remaining part,   b2=a2−n<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>b</mi><mn>2</mn></msup><mo>=</mo><msup><mi>a</mi><mn>2</mn></msup><mo>−</mo><mi>n</mi></math>$b^2 = a^2 - n$ , is also a square number. If it is, then we have found the factors   a−b<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi><mo>−</mo><mi>b</mi></math>$a - b$  and   a+b<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi><mo>+</mo><mi>b</mi></math>$a + b$  of   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ .

    int fermat(int n) {
    int a = ceil(sqrt(n));
    int b2 = a*a - n;
    int b = round(sqrt(b2));
    while (b * b != b2) {
        a = a + 1;
        b2 = a*a - n;
        b = round(sqrt(b2));
    }
    return a - b;
}

    This factorization method can be very fast if the difference between the two factors   p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$  and   q<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>q</mi></math>$q$  is small. The algorithm runs in   O(|p−q|)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mrow data-mjx-texclass="ORD"><mo stretchy="false">|</mo></mrow><mi>p</mi><mo>−</mo><mi>q</mi><mrow data-mjx-texclass="ORD"><mo stretchy="false">|</mo></mrow><mo stretchy="false">)</mo></math>$O(|p - q|)$  time. In practice though, this method is rarely used. Once factors become further apart, it is extremely slow.

    However, there are still a large number of optimization options regarding this approach. By looking at the squares   a2<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>a</mi><mn>2</mn></msup></math>$a^2$  modulo a fixed small number, it can be observed that certain values   a<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi></math>$a$  don't have to be viewed, since they cannot produce a square number   a2−n<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>a</mi><mn>2</mn></msup><mo>−</mo><mi>n</mi></math>$a^2 - n$ .

    Pollard's   p−1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo>−</mo><mn>1</mn></math>$p - 1$  method¶

    It is very likely that at least one factor of a number is   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ -powersmooth for small   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ .   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ -powersmooth means that every prime power   dk<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>d</mi><mi>k</mi></msup></math>$d^k$  that divides   p−1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo>−</mo><mn>1</mn></math>$p-1$  is at most   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ . E.g. the prime factorization of   4817191<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>4817191</mn></math>$4817191$  is   1303⋅3697<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>1303</mn><mo>⋅</mo><mn>3697</mn></math>$1303 \cdot 3697$ . And the factors are   31<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>31</mn></math>$31$ -powersmooth and   16<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>16</mn></math>$16$ -powersmooth respectably, because   1303−1=2⋅3⋅7⋅31<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>1303</mn><mo>−</mo><mn>1</mn><mo>=</mo><mn>2</mn><mo>⋅</mo><mn>3</mn><mo>⋅</mo><mn>7</mn><mo>⋅</mo><mn>31</mn></math>$1303 - 1 = 2 \cdot 3 \cdot 7 \cdot 31$  and   3697−1=24⋅3⋅7⋅11<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>3697</mn><mo>−</mo><mn>1</mn><mo>=</mo><msup><mn>2</mn><mn>4</mn></msup><mo>⋅</mo><mn>3</mn><mo>⋅</mo><mn>7</mn><mo>⋅</mo><mn>11</mn></math>$3697 - 1 = 2^4 \cdot 3 \cdot 7 \cdot 11$ . In 1974 John Pollard invented a method to extracts   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ -powersmooth factors from a composite number.

    The idea comes from Fermat's little theorem. Let a factorization of   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$  be   n=p⋅q<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo>=</mo><mi>p</mi><mo>⋅</mo><mi>q</mi></math>$n = p \cdot q$ . It says that if   a<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>a</mi></math>$a$  is coprime to   p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$ , the following statement holds:

      ap−1≡1(modp)<math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><msup><mi>a</mi><mrow data-mjx-texclass="ORD"><mi>p</mi><mo>−</mo><mn>1</mn></mrow></msup><mo>≡</mo><mn>1</mn><mspace width="1em"></mspace><mo stretchy="false">(</mo><mi>mod</mi><mspace width="0.333em"></mspace><mi>p</mi><mo stretchy="false">)</mo></math>$$a^{p - 1} \equiv 1 \pmod{p}$$ 

    This also means that

      (a(p−1))k≡ak⋅(p−1)≡1(modp).<math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><msup><mrow data-mjx-texclass="ORD"><mrow data-mjx-texclass="INNER"><mo data-mjx-texclass="OPEN">(</mo><msup><mi>a</mi><mrow data-mjx-texclass="ORD"><mo stretchy="false">(</mo><mi>p</mi><mo>−</mo><mn>1</mn><mo stretchy="false">)</mo></mrow></msup><mo data-mjx-texclass="CLOSE">)</mo></mrow></mrow><mi>k</mi></msup><mo>≡</mo><msup><mi>a</mi><mrow data-mjx-texclass="ORD"><mi>k</mi><mo>⋅</mo><mo stretchy="false">(</mo><mi>p</mi><mo>−</mo><mn>1</mn><mo stretchy="false">)</mo></mrow></msup><mo>≡</mo><mn>1</mn><mspace width="1em"></mspace><mo stretchy="false">(</mo><mi>mod</mi><mspace width="0.333em"></mspace><mi>p</mi><mo stretchy="false">)</mo><mo>.</mo></math>$${\left(a^{(p - 1)}\right)}^k \equiv a^{k \cdot (p - 1)} \equiv 1 \pmod{p}.$$ 

    So for any   M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$  with   p−1 | M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo>−</mo><mn>1</mn><mtext>&nbsp;</mtext><mrow data-mjx-texclass="ORD"><mo stretchy="false">|</mo></mrow><mtext>&nbsp;</mtext><mi>M</mi></math>$p - 1 ~|~ M$  we know that   aM≡1<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>a</mi><mi>M</mi></msup><mo>≡</mo><mn>1</mn></math>$a^M \equiv 1$ . This means that   aM−1=p⋅r<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mi>a</mi><mi>M</mi></msup><mo>−</mo><mn>1</mn><mo>=</mo><mi>p</mi><mo>⋅</mo><mi>r</mi></math>$a^M - 1 = p \cdot r$ , and because of that also   p | gcd(aM−1,n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mtext>&nbsp;</mtext><mrow data-mjx-texclass="ORD"><mo stretchy="false">|</mo></mrow><mtext>&nbsp;</mtext><mo data-mjx-texclass="OP" movablelimits="true">gcd</mo><mo stretchy="false">(</mo><msup><mi>a</mi><mi>M</mi></msup><mo>−</mo><mn>1</mn><mo>,</mo><mi>n</mi><mo stretchy="false">)</mo></math>$p ~|~ \gcd(a^M - 1, n)$ .

    Therefore, if   p−1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo>−</mo><mn>1</mn></math>$p - 1$  for a factor   p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$  of   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$  divides   M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ , we can extract a factor using Euclid's algorithm.

    It is clear, that the smallest   M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$  that is a multiple of every   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ -powersmooth number is   lcm(1, 2 ,3 ,4 , …, B)<math xmlns="http://www.w3.org/1998/Math/MathML"><mtext>lcm</mtext><mo stretchy="false">(</mo><mn>1</mn><mo>,</mo><mtext>&nbsp;</mtext><mn>2</mn><mtext>&nbsp;</mtext><mo>,</mo><mn>3</mn><mtext>&nbsp;</mtext><mo>,</mo><mn>4</mn><mtext>&nbsp;</mtext><mo>,</mo><mtext>&nbsp;</mtext><mo>…</mo><mo>,</mo><mtext>&nbsp;</mtext><mi>B</mi><mo stretchy="false">)</mo></math>$\text{lcm}(1,~2~,3~,4~,~\dots,~B)$ . Or alternatively:

      M=∏prime q≤Bq⌊logq⁡B⌋<math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><mi>M</mi><mo>=</mo><munder><mo data-mjx-texclass="OP">∏</mo><mrow data-mjx-texclass="ORD"><mtext>prime&nbsp;</mtext><mi>q</mi><mo>≤</mo><mi>B</mi></mrow></munder><msup><mi>q</mi><mrow data-mjx-texclass="ORD"><mo fence="false" stretchy="false">⌊</mo><msub><mi>log</mi><mi>q</mi></msub><mo data-mjx-texclass="NONE">⁡</mo><mi>B</mi><mo fence="false" stretchy="false">⌋</mo></mrow></msup></math>$$M = \prod_{\text{prime } q \le B} q^{\lfloor \log_q B \rfloor}$$ 

    Notice, if   p−1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo>−</mo><mn>1</mn></math>$p-1$  divides   M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$  for all prime factors   p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$  of   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ , then   gcd(aM−1,n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo data-mjx-texclass="OP" movablelimits="true">gcd</mo><mo stretchy="false">(</mo><msup><mi>a</mi><mi>M</mi></msup><mo>−</mo><mn>1</mn><mo>,</mo><mi>n</mi><mo stretchy="false">)</mo></math>$\gcd(a^M - 1, n)$  will just be   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ . In this case we don't receive a factor. Therefore, we will try to perform the   gcd<math xmlns="http://www.w3.org/1998/Math/MathML"><mo data-mjx-texclass="OP" movablelimits="true">gcd</mo></math>$\gcd$  multiple times, while we compute   M<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>M</mi></math>$M$ .

    Some composite numbers don't have   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ -powersmooth factors for small   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$ . For example, the factors of the composite number   100 000 000 000 000 493=763 013⋅131 059 365 961<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>100</mn><mtext>&nbsp;</mtext><mn>000</mn><mtext>&nbsp;</mtext><mn>000</mn><mtext>&nbsp;</mtext><mn>000</mn><mtext>&nbsp;</mtext><mn>000</mn><mtext>&nbsp;</mtext><mn>493</mn><mo>=</mo><mn>763</mn><mtext>&nbsp;</mtext><mn>013</mn><mo>⋅</mo><mn>131</mn><mtext>&nbsp;</mtext><mn>059</mn><mtext>&nbsp;</mtext><mn>365</mn><mtext>&nbsp;</mtext><mn>961</mn></math>$100~000~000~000~000~493 = 763~013 \cdot 131~059~365~961$  are   190 753<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>190</mn><mtext>&nbsp;</mtext><mn>753</mn></math>$190~753$ -powersmooth and   1 092 161 383<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>1</mn><mtext>&nbsp;</mtext><mn>092</mn><mtext>&nbsp;</mtext><mn>161</mn><mtext>&nbsp;</mtext><mn>383</mn></math>$1~092~161~383$ -powersmooth. We will have to choose   B>=190 753<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi><mo>&gt;=</mo><mn>190</mn><mtext>&nbsp;</mtext><mn>753</mn></math>$B >= 190~753$  to factorize the number.

    In the following implementation we start with   B=10<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi><mo>=</mo><mn>10</mn></math>$B = 10$  and increase   B<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>B</mi></math>$B$  after each each iteration.

    long long pollards_p_minus_1(long long n) {
    int B = 10;
    long long g = 1;
    while (B <= 1000000 && g < n) {
        long long a = 2 + rand() %  (n - 3);
        g = gcd(a, n);
        if (g > 1)
            return g;

        // compute a^M
        for (int p : primes) {
            if (p >= B)
                continue;
            long long p_power = 1;
            while (p_power * p <= B)
                p_power *= p;
            a = power(a, p_power, n);

            g = gcd(a - 1, n);
            if (g > 1 && g < n)
                return g;
        }
        B *= 2;
    }
    return 1;
}

    Observe that this is a probabilistic algorithm. A consequence of this is that there is a possibility of the algorithm being unable to find a factor at all.

    The complexity is   O(Blog⁡Blog2⁡n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>B</mi><mi>log</mi><mo data-mjx-texclass="NONE">⁡</mo><mi>B</mi><msup><mi>log</mi><mn>2</mn></msup><mo data-mjx-texclass="NONE">⁡</mo><mi>n</mi><mo stretchy="false">)</mo></math>$O(B \log B \log^2 n)$  per iteration.

    Pollard's rho algorithm¶

    Pollard's Rho Algorithm is yet another factorization algorithm from John Pollard.

    Let the prime factorization of a number be   n=pq<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo>=</mo><mi>p</mi><mi>q</mi></math>$n = p q$ . The algorithm looks at a pseudo-random sequence   {xi}={x0, f(x0), f(f(x0)), …}<math xmlns="http://www.w3.org/1998/Math/MathML"><mo fence="false" stretchy="false">{</mo><msub><mi>x</mi><mi>i</mi></msub><mo fence="false" stretchy="false">}</mo><mo>=</mo><mo fence="false" stretchy="false">{</mo><msub><mi>x</mi><mn>0</mn></msub><mo>,</mo><mtext>&nbsp;</mtext><mi>f</mi><mo stretchy="false">(</mo><msub><mi>x</mi><mn>0</mn></msub><mo stretchy="false">)</mo><mo>,</mo><mtext>&nbsp;</mtext><mi>f</mi><mo stretchy="false">(</mo><mi>f</mi><mo stretchy="false">(</mo><msub><mi>x</mi><mn>0</mn></msub><mo stretchy="false">)</mo><mo stretchy="false">)</mo><mo>,</mo><mtext>&nbsp;</mtext><mo>…</mo><mo fence="false" stretchy="false">}</mo></math>$\{x_i\} = \{x_0,~f(x_0),~f(f(x_0)),~\dots\}$  where   f<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi></math>$f$  is a polynomial function, usually   f(x)=(x2+c)modn<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi><mo stretchy="false">(</mo><mi>x</mi><mo stretchy="false">)</mo><mo>=</mo><mo stretchy="false">(</mo><msup><mi>x</mi><mn>2</mn></msup><mo>+</mo><mi>c</mi><mo stretchy="false">)</mo><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mi>n</mi></math>$f(x) = (x^2 + c) \bmod n$  is chosen with   c=1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi><mo>=</mo><mn>1</mn></math>$c = 1$ .

    In this instance, we are not interested in the sequence   {xi}<math xmlns="http://www.w3.org/1998/Math/MathML"><mo fence="false" stretchy="false">{</mo><msub><mi>x</mi><mi>i</mi></msub><mo fence="false" stretchy="false">}</mo></math>$\{x_i\}$ . We are more interested in the sequence   {ximodp}<math xmlns="http://www.w3.org/1998/Math/MathML"><mo fence="false" stretchy="false">{</mo><msub><mi>x</mi><mi>i</mi></msub><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mi>p</mi><mo fence="false" stretchy="false">}</mo></math>$\{x_i \bmod p\}$ . Since   f<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi></math>$f$  is a polynomial function, and all the values are in the range   [0; p)<math xmlns="http://www.w3.org/1998/Math/MathML"><mo stretchy="false">[</mo><mn>0</mn><mo>;</mo><mtext>&nbsp;</mtext><mi>p</mi><mo stretchy="false">)</mo></math>$[0;~p)$ , this sequence will eventually converge into a loop. The birthday paradox actually suggests that the expected number of elements is   O(p)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><msqrt><mi>p</mi></msqrt><mo stretchy="false">)</mo></math>$O(\sqrt{p})$  until the repetition starts. If   p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$  is smaller than   n<math xmlns="http://www.w3.org/1998/Math/MathML"><msqrt><mi>n</mi></msqrt></math>$\sqrt{n}$ , the repetition will likely start in   O(n4)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mroot><mi>n</mi><mn>4</mn></mroot><mo stretchy="false">)</mo></math>$O(\sqrt[4]{n})$ .

    Here is a visualization of such a sequence   {ximodp}<math xmlns="http://www.w3.org/1998/Math/MathML"><mo fence="false" stretchy="false">{</mo><msub><mi>x</mi><mi>i</mi></msub><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mi>p</mi><mo fence="false" stretchy="false">}</mo></math>$\{x_i \bmod p\}$  with   n=2206637<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi><mo>=</mo><mn>2206637</mn></math>$n = 2206637$ ,   p=317<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mo>=</mo><mn>317</mn></math>$p = 317$ ,   x0=2<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mn>0</mn></msub><mo>=</mo><mn>2</mn></math>$x_0 = 2$  and   f(x)=x2+1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi><mo stretchy="false">(</mo><mi>x</mi><mo stretchy="false">)</mo><mo>=</mo><msup><mi>x</mi><mn>2</mn></msup><mo>+</mo><mn>1</mn></math>$f(x) = x^2 + 1$ . From the form of the sequence you can see very clearly why the algorithm is called Pollard's   ρ<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>ρ</mi></math>$\rho$  algorithm.

    Yet, there is still an open question. How can we exploit the properties of the sequence   {ximodp}<math xmlns="http://www.w3.org/1998/Math/MathML"><mo fence="false" stretchy="false">{</mo><msub><mi>x</mi><mi>i</mi></msub><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mi>p</mi><mo fence="false" stretchy="false">}</mo></math>$\{x_i \bmod p\}$  to our advantage without even knowing the number   p<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi></math>$p$  itself?

    It's actually quite easy. There is a cycle in the sequence   {ximodp}i≤j<math xmlns="http://www.w3.org/1998/Math/MathML"><mo fence="false" stretchy="false">{</mo><msub><mi>x</mi><mi>i</mi></msub><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mi>p</mi><msub><mo fence="false" stretchy="false">}</mo><mrow data-mjx-texclass="ORD"><mi>i</mi><mo>≤</mo><mi>j</mi></mrow></msub></math>$\{x_i \bmod p\}_{i \le j}$  if and only if there are two indices   s,t≤j<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi><mo>,</mo><mi>t</mi><mo>≤</mo><mi>j</mi></math>$s, t \le j$  such that   xs≡xtmodp<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mi>s</mi></msub><mo>≡</mo><msub><mi>x</mi><mi>t</mi></msub><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mi>p</mi></math>$x_s \equiv x_t \bmod p$ . This equation can be rewritten as   xs−xt≡0modp<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mi>s</mi></msub><mo>−</mo><msub><mi>x</mi><mi>t</mi></msub><mo>≡</mo><mn>0</mn><mo lspace="thickmathspace" rspace="thickmathspace">mod</mo><mi>p</mi></math>$x_s - x_t \equiv 0 \bmod p$  which is the same as   p | gcd(xs−xt,n)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>p</mi><mtext>&nbsp;</mtext><mrow data-mjx-texclass="ORD"><mo stretchy="false">|</mo></mrow><mtext>&nbsp;</mtext><mo data-mjx-texclass="OP" movablelimits="true">gcd</mo><mo stretchy="false">(</mo><msub><mi>x</mi><mi>s</mi></msub><mo>−</mo><msub><mi>x</mi><mi>t</mi></msub><mo>,</mo><mi>n</mi><mo stretchy="false">)</mo></math>$p ~|~ \gcd(x_s - x_t, n)$ .

    Therefore, if we find two indices   s<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>s</mi></math>$s$  and   t<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>t</mi></math>$t$  with   g=gcd(xs−xt,n)>1<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mo>=</mo><mo data-mjx-texclass="OP" movablelimits="true">gcd</mo><mo stretchy="false">(</mo><msub><mi>x</mi><mi>s</mi></msub><mo>−</mo><msub><mi>x</mi><mi>t</mi></msub><mo>,</mo><mi>n</mi><mo stretchy="false">)</mo><mo>&gt;</mo><mn>1</mn></math>$g = \gcd(x_s - x_t, n) > 1$ , we have found a cycle and also a factor   g<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi></math>$g$  of   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$ . It is possible that   g=n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>g</mi><mo>=</mo><mi>n</mi></math>$g = n$ . In this case we haven't found a proper factor, so we must repeat the algorithm with a different parameter (different starting value   x0<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mn>0</mn></msub></math>$x_0$ , different constant   c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$  in the polynomial function   f<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi></math>$f$ ).

    To find the cycle, we can use any common cycle detection algorithm.

    Floyd's cycle-finding algorithm¶

    This algorithm finds a cycle by using two pointers moving over the sequence at differing speeds. During each iteration, the first pointer will advance one element over, while the second pointer advances to every other element. Using this idea it is easy to observe that if there is a cycle, at some point the second pointer will come around to meet the first one during the loops. If the cycle length is   λ<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>λ</mi></math>$\lambda$  and the   μ<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>μ</mi></math>$\mu$  is the first index at which the cycle starts, then the algorithm will run in   O(λ+μ)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>λ</mi><mo>+</mo><mi>μ</mi><mo stretchy="false">)</mo></math>$O(\lambda + \mu)$  time.

    This algorithm is also known as the Tortoise and Hare algorithm, based on the tale in which a tortoise (the slow pointer) and a hare (the faster pointer) have a race.

    It is actually possible to determine the parameter   λ<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>λ</mi></math>$\lambda$  and   μ<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>μ</mi></math>$\mu$  using this algorithm (also in   O(λ+μ)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mi>λ</mi><mo>+</mo><mi>μ</mi><mo stretchy="false">)</mo></math>$O(\lambda + \mu)$  time and   O(1)<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mn>1</mn><mo stretchy="false">)</mo></math>$O(1)$  space). When a cycle is detected, the algorithm will return 'True'. If the sequence doesn't have a cycle, then the function will loop endlessly. However, using Pollard's Rho Algorithm, this can be prevented. 

    function floyd(f, x0):
    tortoise = x0
    hare = f(x0)
    while tortoise != hare:
        tortoise = f(tortoise)
        hare = f(f(hare))
    return true

    First, here is an implementation using the Floyd's cycle-finding algorithm. The algorithm generally runs in   O(n4log⁡(n))<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>O</mi><mo stretchy="false">(</mo><mroot><mi>n</mi><mn>4</mn></mroot><mi>log</mi><mo data-mjx-texclass="NONE">⁡</mo><mo stretchy="false">(</mo><mi>n</mi><mo stretchy="false">)</mo><mo stretchy="false">)</mo></math>$O(\sqrt[4]{n} \log(n))$  time.

    long long mult(long long a, long long b, long long mod) {
    return (__int128)a * b % mod;
}

long long f(long long x, long long c, long long mod) {
    return (mult(x, x, mod) + c) % mod;

long long rho(long long n, long long x0=2, long long c=1) {
    long long x = x0;
    long long y = x0;
    long long g = 1;
    while (g == 1) {
        x = f(x, c, n);
        y = f(y, c, n);
        y = f(y, c, n);
        g = gcd(abs(x - y), n);
    }
    return g;
}

If GCC is not available, you can using a similar idea as binary exponentiation.

    long long mult(long long a, long long b, long long mod) {
    long long result = 0;
    while (b) {
        if (b & 1)
            result = (result + a) % mod;
        a = (a + a) % mod;
        b >>= 1;
    }
    return result;
}

    Alternatively you can also implement the Montgomery multiplication.

    As stated previously, if   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$  is composite and the algorithm returns   n<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>n</mi></math>$n$  as factor, you have to repeat the procedure with different parameters   x0<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mn>0</mn></msub></math>$x_0$  and   c<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi></math>$c$ . E.g. the choice   x0=c=1<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mn>0</mn></msub><mo>=</mo><mi>c</mi><mo>=</mo><mn>1</mn></math>$x_0 = c = 1$  will not factor   25=5⋅5<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>25</mn><mo>=</mo><mn>5</mn><mo>⋅</mo><mn>5</mn></math>$25 = 5 \cdot 5$ . The algorithm will return   25<math xmlns="http://www.w3.org/1998/Math/MathML"><mn>25</mn></math>$25$ . However, the choice   x0=1<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mn>0</mn></msub><mo>=</mo><mn>1</mn></math>$x_0 = 1$ ,   c=2<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>c</mi><mo>=</mo><mn>2</mn></math>$c = 2$  will factor it.

    Brent's algorithm¶

    Brent implements a similar method to Floyd, using two pointers. The difference being that instead of advancing the pointers by one and two places respectively, they are advanced by powers of two. As soon as   2i<math xmlns="http://www.w3.org/1998/Math/MathML"><msup><mn>2</mn><mi>i</mi></msup></math>$2^i$  is greater than   λ<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>λ</mi></math>$\lambda$  and   μ<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>μ</mi></math>$\mu$ , we will find the cycle.

    function floyd(f, x0):
    tortoise = x0
    hare = f(x0)
    l = 1
    while tortoise != hare:
        tortoise = hare
        repeat l times:
            hare = f(hare)
            if tortoise == hare:
                return true
        l *= 2
    return true

    Brent's algorithm also runs in linear time, but is generally faster than Floyd's, since it uses less evaluations of the function   f<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>f</mi></math>$f$ .

    Implementation¶

    The straightforward implementation of Brent's algorithm can be sped up by omitting the terms   xl−xk<math xmlns="http://www.w3.org/1998/Math/MathML"><msub><mi>x</mi><mi>l</mi></msub><mo>−</mo><msub><mi>x</mi><mi>k</mi></msub></math>$x_l - x_k$  if   k<3⋅l2<math xmlns="http://www.w3.org/1998/Math/MathML"><mi>k</mi><mo>&lt;</mo><mfrac><mrow><mn>3</mn><mo>⋅</mo><mi>l</mi></mrow><mn>2</mn></mfrac></math>$k < \frac{3 \cdot l}{2}$ . In addition, instead of performing the   gcd<math xmlns="http://www.w3.org/1998/Math/MathML"><mo data-mjx-texclass="OP" movablelimits="true">gcd</mo></math>$\gcd$  computation at every step, we multiply the terms and only actually check   gcd<math xmlns="http://www.w3.org/1998/Math/MathML"><mo data-mjx-texclass="OP" movablelimits="true">gcd</mo></math>$\gcd$  every few steps and backtrack if overshot.

    long long brent(long long n, long long x0=2, long long c=1) {
    long long x = x0;
    long long g = 1;
    long long q = 1;
    long long xs, y;

    int m = 128;
    int l = 1;
    while (g == 1) {
        y = x;
        for (int i = 1; i < l; i++)
            x = f(x, c, n);
        int k = 0;
        while (k < l && g == 1) {
            xs = x;
            for (int i = 0; i < m && i < l - k; i++) {
                x = f(x, c, n);
                q = mult(q, abs(y - x), n);
            }
            g = gcd(q, n);
            k += m;
        }
        l *= 2;
    }
    if (g == n) {
        do {
            xs = f(xs, c, n);
            g = gcd(abs(xs - y), n);
        } while (g == 1);
    }
    return g;
}

    The combination of a trial division for small prime numbers together with Brent's version of Pollard's rho algorithm makes a very powerful factorization algorithm.
